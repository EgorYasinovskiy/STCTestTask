## Общее описание 
Тестовое задание. Маршрутизатор запросов на C#. Регистрирует шаблоны путей с
динамическими сегментами (`/users/{id:int}/`) и делегаты-обработчики, и умеет
по входной строке находить подходящий обработчик, парсить значения и вызывать
его.
Поддерживает как синхронные делегаты а также передачу
CancellationToken в обработчик. Построен на древовидной структуре, поэтому
поиск маршрута — это O(кол-во сегментов), без перебора всех зарегистрированных
маршрутов.

## Описание

Структура данных для хранение маршрутов - дерево из узлов RouteNode
Роутер хранит все зарегистрированные шаблоны в дереве. Каждый узел
соответствует одному сегменту пути и хранит
- StaticChildren — словарь узлов по имени сегмента
  (например, `users` или `posts`).
- DynamicChildren — словарь узлов по типу динамического
  сегмента. Это позволяет одновременно держать `/x/{v:int}/` и
  `/x/{v:guid}/` — у них разные типы, и мы разводим их по разным "веткам".
- Endpoint — сам обработчик, лежит в узле, который соответствует
  последнему сегменту шаблона.

Дерево позволяет находить обработчик за O(длина пути): на каждом сегменте
мы сначала пробуем точное статическое совпадение, и только если его нет
перебираем динамические узлы, пытаясь распарсить сегмент под
каждый из зарегистрированных типов. (tryparse достаточно "дорогой" в плане операции, но это позволяет регистрировать несколько типов сразу)

Поиск с возвратом даёт приоритет статике: если есть и
`/foo/static/`, и `/foo/{name:string}/`, то запрос `/foo/static/` попадёт в
статический обработчик, а не будет спаршен динамически.

### Класс `Endpoint`

При регистрации маршрута "/users/{id:int}/",
роутер должен при вызове "/users/42/")" взять строку "42", распарсить
её в число и передать в делегат. Помимо самого значения нужно знать,
в какую позицию делегата его положить (если у делегата несколько параметров).

Endpoint создаётся один раз при регистрации маршрута и хранит:

- `Action` - сам делегат, внутри Delegate без типа, чтобы отказаться от бесконечного количества перегрузок из примера Action<T1,T2,T3....>
- `ParamNames` - имена параметров делегата, извлеченные через рефлексию.
- `_segmentToArgIndex` - массив той же длины, что и число динамических
  сегментов. Хранит "словарь" позиции аргумента в шаблоне и в делегате. ключ - позиция элемента в словаре, значение - актуальный индекс.
  Для foo/{a:int}/bar/{b:int} и  делегата (int b, int a)=>return a+b; будет хранить значение [1,0]
- `IsAsync` - флаг: возвращает ли делегат `Task` или `Task<T>`. Если да,
  роутер после `DynamicInvoke` дожидается `Task` через `await`.
- `TakesCancellationToken` - флаг: является ли последний параметр делегата
  типом `CancellationToken`. Если да, при вызове туда автоматически
  подставляется токен отмены.

В конструкторе Endpoint ещё делает валидацию
- Каждый сегмент шаблона должен соответствовать какому-то параметру делегата
  по имени. Иначе ArgumentException.
- Тип сегмента должен быть совместим с типом параметра делегата.
  `{a:int}` нельзя забиндить на `(string a) => {...}'.
- У делегата не должно быть лишних параметров, для которых нет сегмента
  в шаблоне (кроме необязательного `CancellationToken`).

### Поиск и вызов маршрута (`Route` / `RouteAsync`)

При вызове `Route("/users/42/")` выполняется следующий порядок действий.

1. Строка разбивается на сегменты по слешу ("users", "42").
2. Начиная с корневого элемента, идём по сегментам. На каждом шаге:
   - сначала ищем статический дочерний узел с таким же именем;
   - если не нашли, то перебираем динамические дочерние узлы, пытаемся
     распарсить сегмент в нужный тип; если получилось, спускаемся
     в этот дочерний узел.
3. Когда сегменты кончились и в узле есть `Endpoint`, то мы нашли
   обработчик. Сохраняем распарсенные значения по пути.
4. `Endpoint.BuildArgs` расставляет значения в правильные позиции
   аргументов делегата (по `_segmentToArgIndex`).
5. `Delegate.DynamicInvoke(args)` вызывает делегат.
6. Если делегат `async`, роутер получает `Task` и эвейтит его.
   Любое исключение из `DynamicInvoke` разворачивается через
   `ExceptionDispatchInfo`, чтобы наружу не вылезал служебный
   `TargetInvocationException`.

Если маршрут не найден — `InvalidOperationException`. Если делегат
бросил исключение пробрасывается как есть.
